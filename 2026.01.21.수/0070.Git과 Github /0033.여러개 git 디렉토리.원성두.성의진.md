## Git 저장소(Repository)의 구조와 관리

Git을 설치한 후 프로젝트별로 디렉토리를 생성하고 Git 설정을 초기화하면, 각 디렉토리는 독립적인 **로컬 레포지토리(Local Repository)**가 됩니다. 이에 대한 핵심 개념을 정리해 드립니다.

---

### 1. 개별 디렉토리의 레포지토리화

단순히 폴더(디렉토리)를 만든다고 해서 바로 레포지토리가 되는 것은 아닙니다. 해당 디렉토리에서 Git 초기화 명령을 실행해야 합니다.

* **명령어:** `git init`
* **결과:** 명령어를 실행한 디렉토리 안에 `.git`이라는 숨겨진 폴더가 생성됩니다.
* **독립성:** 이 `.git` 폴더가 생성된 시점부터 해당 디렉토리는 하나의 독립된 **레포지토리**가 됩니다. A 프로젝트 폴더와 B 프로젝트 폴더에 각각 `git init`을 했다면, 두 폴더의 이력은 서로 전혀 영향을 주지 않고 별개로 관리됩니다.

### 2. 로컬 레포지토리와 원격 레포지토리의 관계

실무에서는 내 컴퓨터에 있는 '로컬 레포지토리'와 GitHub/GitLab 같은 서비스에 있는 '원격 레포지토리'를 연결하여 사용합니다.

* **로컬 레포지토리(Local):** 내 PC의 특정 디렉토리에서 이루어지는 모든 변경 이력(Commit)이 저장되는 곳입니다.
* **원격 레포지토리(Remote):** 서버에 위치하며, 협업을 위해 로컬의 작업 내용을 `push`하거나 다른 사람의 내용을 `pull` 받는 공간입니다. 보통 **1개의 로컬 레포지토리는 1개의 원격 레포지토리와 연결**하는 것이 일반적입니다.

초보자가 가장 많이 하는 실수 중 하나는 **Git 레포지토리 안에 또 다른 Git 레포지토리를 만드는 것**입니다.

* 상위 폴더(예: `Documents`)에서 `git init`을 한 상태인데, 그 하위 폴더(예: `Project_A`)에서 다시 `git init`을 하면 관리가 매우 복잡해집니다.
* **권장 사항:** 항상 실제 코드가 들어가는 최상위 프로젝트 폴더 하나당 하나의 `git init`만 수행하십시오.

---

### [요약] 프로젝트 생성 루틴

1. 프로젝트 폴더 생성: `mkdir my-project`
2. 폴더 진입: `cd my-project`
3. **Git 초기화(레포지토리 생성): `git init**`
4. 작업 후 커밋 및 원격 연결(`git remote add origin ...`)

## Git 레포지토리와 origin의 범위

Git의 설정 범위와 `origin`이라는 별칭의 유효 범위를 이해하는 것은 다중 프로젝트 관리의 핵심입니다.

---

### 1. origin은 광역 변수인가?

결론부터 말씀드리면, `origin`은 광역 변수(Global Variable)가 아니라 **로컬 레포지토리별 지역 변수(Local Variable)**에 가깝습니다.

* **범위(Scope):** `origin`이라는 별칭은 특정 디렉토리 안에 생성된 `.git` 폴더 내부에만 저장됩니다.
* **독립성:** 프로젝트 A 폴더에서 정의한 `origin`과 프로젝트 B 폴더에서 정의한 `origin`은 이름만 같을 뿐, 각각 서로 다른 URL 주소를 가리키는 별개의 설정입니다.

---

### 2. 프로젝트별 독립적인 origin 구성

여러 개의 프로젝트(레포지토리)를 운영할 때 각 저장소는 자신만의 원격 연결 정보를 가집니다.

| 프로젝트 폴더 | 로컬 저장소 위치 | origin이 가리키는 곳 (예시) |
| --- | --- | --- |
| **Project_A** | `/Users/work/A/.git` | `https://github.com/user/repo-A.git` |
| **Project_B** | `/Users/work/B/.git` | `https://github.com/user/repo-B.git` |
| **Project_C** | `/Users/work/C/.git` | `https://gitlab.com/user/repo-C.git` |

위 표와 같이, 각 폴더로 이동(`cd`)하여 `git remote -v` 명령어를 입력해 보면 각 프로젝트에 설정된 서로 다른 원격 주소를 확인할 수 있습니다.

---

### 3. 왜 모든 프로젝트에서 origin이라는 이름을 쓰나요?

모든 프로젝트에서 각기 다른 원격 주소를 쓰면서도 이름을 `origin`으로 통일하는 이유는 **워크플로우의 표준화** 때문입니다.

* 만약 프로젝트 A에서는 별칭을 `github_a`로 하고, 프로젝트 B에서는 `my_server`로 한다면, 매번 푸시할 때마다 `git push github_a main`, `git push my_server main`처럼 명령어를 다르게 입력해야 합니다.
* 모든 프로젝트에서 별칭을 **`origin`**으로 통일하면, 어떤 폴더에 있든 항상 **`git push origin main`**이라는 동일한 명령어로 작업할 수 있어 실수를 줄이고 생산성을 높일 수 있습니다.

---

### 4. 핵심 요약

1. **독립성:** 각 Git 레포지토리는 고유한 `.git` 설정 파일을 가지며, 그 안의 `origin` 설정은 해당 레포지토리 내에서만 유효합니다.
2. **중복 가능:** 프로젝트가 100개라면 100개 각각의 폴더에 서로 다른 주소를 가진 `origin` 별칭을 만들 수 있습니다.
3. **관리:** `origin`은 전역 설정이 아니므로 한 프로젝트에서 `origin` 주소를 변경해도 다른 프로젝트에는 전혀 영향을 주지 않습니다.
