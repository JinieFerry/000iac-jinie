## Remote Repository: origin의 이해

Git에서 **`origin`**은 사용자가 복제(clone)해온 **원격 저장소(Remote Repository)에 부여된 기본 별칭(Alias)**입니다. 단순히 긴 URL(예: `https://github.com/user/project.git`)을 매번 입력하는 번거로움을 피하기 위해 사용하는 일종의 단축어라고 이해할 수 있습니다.

---

### 1. origin의 주요 역할

* **기본 연결점:** `git clone [URL]` 명령어를 통해 저장소를 복제하면, Git은 자동으로 해당 URL을 `origin`이라는 이름으로 등록합니다.
* **업로드 및 다운로드 대상:** 로컬에서 작업한 내용을 서버에 올리거나(`push`), 서버의 변경 사항을 내려받을 때(`pull`) 대상 저장소를 가리키는 이름으로 사용됩니다.
* **다중 원격 저장소 관리:** 필요에 따라 `origin` 외에도 `upstream`, `backup` 등 다른 별칭을 추가하여 여러 원격 저장소를 관리할 수 있습니다.

---

### 2. 관련 핵심 명령어

원격 저장소 `origin`을 관리하고 확인하는 데 사용되는 주요 명령어들입니다.

| 명령어 | 설명 | 비고 |
| --- | --- | --- |
| `git remote -v` | 현재 연결된 원격 저장소의 이름과 URL 확인 | `fetch`, `push` 주소 표시 |
| `git remote add origin [URL]` | 새로운 원격 저장소를 `origin`이라는 이름으로 추가 | `git init` 후 최초 연결 시 사용 |
| `git remote set-url origin [URL]` | 기존 `origin`의 원격 저장소 주소를 변경 | 저장소 이전 시 사용 |
| `git remote show origin` | 원격 저장소의 상세 정보 및 브랜치 상태 확인 | 로컬/원격 브랜치 매핑 정보 |
| `git remote rename origin [new_name]` | `origin`이라는 별칭을 다른 이름으로 변경 | - |

---


### 3. origin 활용 워크플로우

일반적으로 로컬 저장소와 원격 저장소(`origin`) 사이에서 데이터는 다음과 같이 흐릅니다.

1. **연결 상태 확인**
```bash
git remote -v
# 결과 예시: origin https://github.com/username/repo.git (fetch)

```
# 전에는 apt update도 없었음 직접 사이트 가서 다운로드 받아야 했음 (어디가면 뭐 받을 수 있는지를 자기가 다 가지고 있고 버전 업데이트 되면 바뀌는 주소를 계속 다시 받는 것)
# 까먹지 말아야 함 중요 -^0 
2. **로컬 작업을 원격에 반영 (Push)**
```bash
git pull origin main  # 먼저 origin의 변경사항 반영 #complict 방지 내 안에 저장 해 둔 것 다운로드 받는 것 
git push origin main
# 로컬의 main 브랜치 내용을 origin(원격)의 main 브랜치로 업로드
git config pull.rebase true  # 안되면 rebase로 origin을 먼저 수용한후 내 작업 수행하겠음. #rebase:base는 베이스인데 다시 베이스를 잡는다. 풀을 먼저 해두고 푸시하는 것

```
git config가 들어오면 설정을 아예 해두는 것-> true 로 해두면 최초 1회만 하면 됨
push 전에 항상 pull을 하기 -^0


3. **원격의 최신 상태를 로컬로 가져오기 (Pull)**
```bash
git pull origin main
# origin(원격)의 최신 이력을 가져와서 로컬 브랜치와 병합

```

### 4. AWS 환경에서의 origin (CodeCommit)

**AWS CodeCommit**을 사용하여 프로젝트를 관리할 때도 동일한 메커니즘이 적용됩니다.

* **연결:** `git remote add origin https://git-codecommit.ap-northeast-2.amazonaws.com/v1/repos/MyRepo`
* **인증:** AWS CLI의 `credential-helper`를 사용하면 별도의 ID/PW 입력 없이 `origin`과 통신할 수 있습니다.
* **비용 유의:** * **Data Transfer Out:** `origin`에서 로컬로 데이터를 대량으로 `pull` 하거나 `clone` 할 때, 리전 외부로 데이터가 전송되면 AWS 데이터 전송 비용(GB당 과금)이 발생할 수 있습니다.
* **API 요청:** Git push/pull 시 발생하는 API 요청 횟수가 무료 티어 범위를 초과하면 과금 대상이 됩니다.



---

### 5. 요약: 왜 origin인가?

Git은 분산 버전 관리 시스템이므로, 반드시 하나의 서버만 가질 필요는 없습니다. 하지만 혼란을 방지하기 위해 **"내가 가져온 본래의 근원지"**라는 의미로 `origin`이라는 약속된 이름을 관례적으로 사용하는 것입니다.
